# Plan: 建立格式 Profile 管理能力

## 背景与动机

Profile 是该产品的核心资产，一套 profile 代表一组可复用、可执行、可质检的 Word 格式规范。当前产品方案已经要求支持系统内置、用户手动创建、自然语言生成、格式要求文档生成和示例模板反推等来源，因此需要先定义稳定的 profile schema、版本机制和可视化编辑入口。
<!-- 下游：proposal.md 的 motivation -->

## Goal

- 建立机器可执行、可版本化、可校验的 profile schema，覆盖页面、字体、正文、标题、摘要、目录、页码、图表、公式、参考文献和质检配置。
- 内置“华东师范大学毕业论文格式”示例 profile，使用户能在 Web 端查看、编辑、保存和导出 profile 版本。
<!-- 下游：proposal.md 的 scope -->

## Non-goals

- 不实现 Agent 自动抽取 profile；本 change 只支持手动创建、编辑、导入/导出和内置示例。
- 不实现实际文档重排和质检执行。
- 不实现复杂组织级模板权限、市场化模板库或多人协同审批。
<!-- 下游：proposal.md 的 scope -->

## 当前仓库事实

- 产品方案明确 Profile 是核心资产，每个 profile 代表一套可复用格式规范：`docs/word-format-agent-web-product-plan.md:140`、`docs/word-format-agent-web-product-plan.md:142`。
- 产品方案列出 profile 来源包括系统内置、用户手动创建、自然语言生成、格式要求文档生成和示例 Word 模板反推：`docs/word-format-agent-web-product-plan.md:144`、`docs/word-format-agent-web-product-plan.md:150`。
- 产品方案定义 profile 状态包括 `draft`、`active`、`archived`，并要求每次修改生成新版本、排版任务绑定具体版本：`docs/word-format-agent-web-product-plan.md:152`、`docs/word-format-agent-web-product-plan.md:160`、`docs/word-format-agent-web-product-plan.md:161`。
- 产品方案要求 Web 端提供结构化 profile 编辑表单，覆盖页面、字体、标题、目录、页码、图表题注、公式、参考文献和检查强度，并提供高级导入/导出 `profile.yaml`：`docs/word-format-agent-web-product-plan.md:192`、`docs/word-format-agent-web-product-plan.md:194`、`docs/word-format-agent-web-product-plan.md:206`。
- 产品方案已有 ECNU profile YAML 示例，包含页面、字体、正文、标题、摘要、目录、页码、表格、图、公式、参考文献和 quality 配置：`docs/word-format-agent-web-product-plan.md:396`、`docs/word-format-agent-web-product-plan.md:400`、`docs/word-format-agent-web-product-plan.md:517`。
- 本地 ECNU 格式样本要求 A4、页边距上 2.5cm、下 2.0cm、左 3.0cm、右 2.5cm，装订线居左：`格式集/华东师范大学毕业论文格式要求.doc:9`、`格式集/华东师范大学毕业论文格式要求.doc:13`。
- 本地 ECNU 格式样本要求正文首行缩进 2 字符、1.5 倍行距、英文 Times New Roman：`格式集/华东师范大学毕业论文格式要求.doc:14`、`格式集/华东师范大学毕业论文格式要求.doc:17`。
- 本地 ECNU 格式样本要求摘要 300-500 字、中文宋体五号，正文宋体小四、标题黑体小四、中文题名黑体小三：`格式集/华东师范大学毕业论文格式要求.doc:28`、`格式集/华东师范大学毕业论文格式要求.doc:35`。
- 现有 OpenSpec specs 基线：未验证；补证路径为初始化 OpenSpec 后读取 `openspec/specs/` 中与 profile schema 和 template management 相关的 `spec.md`。

## 改动边界

- 新增 profile schema 定义、验证器和序列化/反序列化能力。
- 新增内置 ECNU profile 文件或种子数据。
- 新增 profile CRUD API、版本 API、导入/导出 API。
- 新增前端 profile 列表、详情、结构化编辑器和 YAML 高级模式。
- 可能需要新增或修改 OpenSpec specs 领域：`format-profile`、`profile-versioning`、`profile-editor`。
<!-- 下游：proposal.md scope，design.md scope，spec deltas 范围 -->

## 约束

- Profile 不能只是提示词，必须是确定性配置，能被排版引擎和质检引擎消费。
- Profile 修改必须产生新版本，历史排版任务必须可追溯到具体 profile 版本。
- ECNU 示例 profile 应基于仓库内格式样本和产品方案，不得把未确认规则伪装成学校事实。
- Web 端应避免让普通用户必须直接编辑 YAML；YAML 仅作为高级模式。
<!-- 下游：design.md 的 constraints -->

## 验收标准

1. 系统内置 `ecnu_thesis` profile，字段覆盖页面、字体、正文、标题、摘要、目录、页码、表格、图、公式、参考文献和 quality 配置。
2. 前端 profile 列表能展示 profile 名称、状态、当前版本、更新时间和来源。
3. 用户能在 Web 端创建 draft profile，编辑页面边距、正文样式、标题样式、图表题注和质检规则，并保存为新版本。
4. 用户能导入合法 YAML 生成 profile draft；导入非法 YAML 时显示字段级错误，不保存无效配置。
5. 用户能导出指定 profile 版本为 YAML。
6. 排版任务创建接口能引用 `profile_id + profile_version`，而不是只引用 profile 名称。
<!-- 下游：spec deltas 的 Scenarios，tasks.md 的 verification -->

## 验证方式

- 运行 profile schema 单元测试，覆盖合法 ECNU profile、缺失必填字段、非法枚举、非法数值范围和未知字段。
- 运行 API 测试，覆盖创建、读取、更新版本、归档、导入、导出和版本引用。
- 手工在 Web 端编辑 ECNU profile，保存后确认版本号变化，旧版本仍可查看。
- 检查导出的 YAML 可再次导入并保持语义一致。
<!-- 下游：tasks.md 的验证步骤 -->

## 迁移 / 回滚 / 降级

- 若 profile 持久化依赖数据库，需要新增 profile 与 profile_versions 相关迁移；回滚时保留导出的 YAML 备份或删除新增表。
- Schema 升级必须考虑旧 profile 版本兼容；不能让历史任务因 schema 新增字段失效。
- 当 profile 校验失败时，应拒绝排版任务并显示原因，而不是降级为默认格式。
