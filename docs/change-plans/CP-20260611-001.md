## Requirement
将当前 Word/PDF 自定义格式导出系统升级为生产级导出链路：用户可通过可视化配置、Agent 对话或上传格式要求文档生成标准 JSON Profile v2，选择 Profile 和可选模板后上传 Word 文档，后端通过确定性 FormatCompiler 生成候选 DOCX，内部 QC 校验与自动修正全部通过后才发布最终 DOCX/PDF 下载。

## Current Behavior
当前文档导出引擎支持 `.doc/.docx` 输入、按 Profile 格式化 DOCX、可选 PDF 导出和文件登记；单文档链路由 `DocumentFormattingService.format_job` 编排，批量链路当前仍会创建质量报告并尝试 fix loop，后续需要改成内部 QC 交付门。

Profile 管理模块当前定义 v1 `FormatProfile`，覆盖页面、字体、正文、标题、摘要、图表题注、公式、参考文献、页眉页脚和基础交付设置，但还不足以稳定表达模板页面、分节、多级编号、固定页、slot 绑定和完整证据链。

Agent 需求与规则抽取模块当前能从自然语言或 `.doc/.docx` 格式要求文档调用 LLM 生成 draft，并要求 LLM 未配置时 fail closed；但输出尚未统一约束为 Profile v2、evidence、missing_fields、unsupported_rules 和后续追问闭环。

内部 QC 模块当前以用户可见 `quality_reports` API 形式提供检查、修复计划和修复执行；升级后需要收敛为内部交付门，不再产出用户可下载的质量报告。

前端工作台当前集中在 `frontend/src/App.tsx`，包含 Profile、Agent 会话、规则文件上传、文档上传、输出下载和质量/修复相关状态；页面需要重构成更清晰的双入口 Profile 创建、模板绑定、导出控制台和最终下载体验。

## Expected Behavior
完成后，系统应形成统一生产链路：

- 可视化规则配置、Agent 对话和上传格式文档三种入口都生成同一套标准 JSON Profile v2。
- Agent 对每条格式规则输出来源证据、置信度、缺失字段和不支持项；LLM 不可用或无法解析时直接报错，不生成假规则。
- Profile v2 支持页面、正文、标题、多级编号、分节、页眉页脚、题注、公式、参考文献、模板绑定、交付门和导出策略等结构化规则。
- 导出只能通过同一个 FormatCompiler 入口执行，API、脚本、批量任务和测试不得分叉为多套格式化逻辑。
- 模板 DOCX 负责固定封面、声明页、目录页、页眉页脚结构和内容 slot；Profile 负责格式规则，输入文档负责正文内容。
- 候选 DOCX 必须经过内部 QC：格式规则、模板适配、DOCX 健康、PDF 可导出性全部通过后才发布最终 DOCX/PDF。
- 用户不再看到质量报告产品概念；前端只展示导出状态、可下载文件、失败原因和需要用户补充的信息。

## Route
Docs Driven

## Route Reason
该需求修改核心导出 workflow、Profile schema、Agent 输出契约、模板系统、内部 QC 行为、后端 API 响应和前端主流程，并影响多个模块长期技术认知，因此必须走 Docs Driven。

## Scope
- `document-export-engine`：引入 FormatCompiler、模板加载、候选 DOCX、内部 QC 放行和最终 DOCX/PDF 发布链路。
- `profile-management`：升级 Profile v2 schema、版本兼容、导入导出、seed profile 和校验能力。
- `agent-requirement-intake`：将自然语言和格式文档抽取统一为 Profile v2 draft，保留 evidence、missing_fields、unsupported_rules 和追问状态。
- `internal-qc-delivery-gate`：把当前质量报告/修复能力收敛为内部 QC 交付门，保留机器校验和自动修正，但不提供用户可下载报告。
- `frontend-workbench`：重构页面为双入口 Profile 创建、Profile 命名沉淀、模板绑定、文档上传、导出进度和 DOCX/PDF 下载工作流。
- `backend-api-shell`：调整跨模块 API 模型、job/batch 状态、导出结果响应和错误表达。
- `file-storage-runtime`：保存模板、候选文件、最终文件、manifest 和必要元数据。
- `dev-docs-ops`：更新模块 docs、OpenSpec artifacts、README/启动说明和必要脚本说明。

## Out Of Scope
- 不在本轮把生产数据库、Redis 队列或对象存储迁移作为必须项；除非后续 OpenSpec 明确纳入。
- 不提供用户可下载的 QC/质量报告；QC 只作为内部放行门。
- 不把前端 JS 生成 DOCX 作为核心导出路径；浏览器端只负责配置和交互，权威导出在后端。
- 不做任意语义改写、论文内容润色或自动补写正文。
- 不承诺对 schema 无法表达、引擎无法验证的规则显示为“合规完成”；这类规则必须进入 unsupported 或 manual review 状态。

## Acceptance
- 用户可通过可视化配置、Agent 对话、上传格式要求文档三种入口之一创建 Profile v2 draft，并在确认后保存为命名 Profile。
- Agent 入口必须调用 LLM；当 `LLM_API_KEY`、`LLM_MODEL` 或 provider 调用不可用时，接口返回明确失败，不得用本地假数据生成规则。
- 上传格式文档抽取出的 Profile v2 至少包含 evidence、missing_fields、unsupported_rules，并能覆盖正文颜色、字体字号、标题层级、编号、页眉页脚、页边距、题注、参考文献和模板绑定相关字段。
- 用户选择 Profile 和可选模板，上传 `.doc/.docx` 后，导出任务通过统一 FormatCompiler 生成候选 DOCX。
- 模板绑定能表达固定封面/声明/目录页、页眉页脚继承和正文内容 slot，导出后不能出现未替换占位符、重复固定页或异常空白页。
- 候选 DOCX 的内部 QC 必须验证 Profile 规则、模板适配、DOCX 可解析、字段刷新、PDF 可导出和 PDF 文本可抽取；全部通过后才写入最终 output file ids。
- 当内部 QC 失败且自动修正仍不能通过时，job/batch 进入失败或需要用户补充状态，并返回简短失败原因；前端不得展示“合规完成”或开放最终下载。
- 前端页面应呈现清晰的四步主流程：创建/选择 Profile、绑定模板、上传待处理文档、下载最终 DOCX/PDF，并移除用户可见质量报告与修复报告概念。
- 后端测试覆盖简单文档、复杂论文样式文档、模板绑定文档、LLM 不可用失败路径、内部 QC 阻断路径和 DOCX/PDF 最终导出路径。
- 前端构建通过，且手工浏览器 smoke 能完成 Profile 创建、文档上传、导出状态查看和最终下载。

## Validation
- `cd backend && uv run pytest`
- `cd frontend && npm run build`
- `openspec validate <change-id> --strict --no-interactive`
- 使用样本文档执行后端 smoke：格式要求文档生成 Profile v2，Profile + 模板 + 待处理 DOCX 生成最终 DOCX/PDF。
- 使用 `codex-docx-inspect <output.docx>` 检查最终 DOCX 段落、表格、字体/字号、页眉页脚和可解析性。
- 使用 `codex-docx-to-pdf <output.docx> <output_dir>` 与 `codex-pdf-inspect <output.pdf>` 检查 PDF 页数、文本可抽取性和明显空白页。
- 浏览器手工验证前端：三种 Profile 入口、Profile 命名保存、模板选择、文档上传、导出状态、最终 DOCX/PDF 下载。

## Risks
- Word OOXML 的复杂结构可能导致部分模板或历史文档无法稳定自动处理，需要以 unsupported/manual review 明确边界。
- LLM 抽取质量会受格式文档表达影响，必须用 schema 校验、evidence 和 deterministic guard 限制不可靠输出。
- Profile v2 会影响已有 seed、API 类型和测试，需要兼容 v1 或提供迁移策略。
- 前端从单文件工作台重构为多工作流页面，容易引入状态遗漏，需要用构建和浏览器 smoke 验证核心路径。
- 内部 QC 从用户可见报告转为交付门时，需要谨慎迁移现有 quality_reports 相关 API/类型，避免旧批量任务逻辑残留。
